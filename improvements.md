# Bundler Improvements

Potential bugs and improvement areas found in `src/platform/server/pack/list/bundler.ts`:

1. **Remote export detection is regex-based and fragile**
   - Can match exports inside comments or strings.
   - Misses valid exports such as `export { foo }`, `export { foo as bar }`, `export const foo: Type = async () => {}`, and `export { foo as default }`.
   - This can make generated client stubs differ from the actual server exports.

2. **Remote function stubs ignore HTTP errors** — ✅ FIXED
   - The generated client stub now checks `res.ok` and throws a reconstructed `Error` (with `name`/`message` from the server's JSON error body) on non-`2xx` responses.
   - Non-JSON error bodies fall back to `"Remote function \"<name>\" failed (HTTP <status>)"`.
   - Residual gap: network-level fetch failures (`TypeError: Failed to fetch`) are not yet wrapped into a `RemoteFunctionError`.

3. **Remote functions returning `undefined` can break client parsing** — ✅ FIXED
   - The server now returns `204 No Content` for `undefined`/`void` returns, and the client stub resolves `undefined` without parsing.
   - Unserializable returns (`function`, `Symbol`, `BigInt`, circular references) are now detected and produce a structured `500` with `name: "RemoteFunctionSerializationError"`.
   - See `skills/huuma-ui-remote-functions/SKILL.md` for the full return-value table.

4. **Bundle hash may be nondeterministic**
   - The bundle hash is generated by concatenating `result.outputFiles` hashes in esbuild's output order.
   - If output ordering changes, identical build content could produce a different overall hash.
   - Sorting by output path before hashing would be safer.

5. **Output file lookup uses only basename**
   - `result.metafile?.outputs[parse(file.path).base]` and `files.set(parse(file.path).base, ...)` rely on basenames.
   - This can break or overwrite files if esbuild emits nested paths or two outputs share the same basename.

6. **Entry-point classification relies on basename**
   - `entryPoints[parse(file.path).name]` assumes the output basename maps directly to the original entry-point key.
   - If an entry-point name contains path separators or esbuild changes output naming, `isEntryPoint`, `isIsland`, or `isRuntime` may be wrong.

7. **All esbuild outputs are treated as script-like files**
   - If esbuild emits CSS or file-loader assets, the bundler returns them through the same output path.
   - The pack layer later serves these via `addScript`, likely with `application/javascript`.
   - This can mishandle CSS, images, fonts, or other non-JS assets imported from client code.

8. **`Bundler.stop()` is global and not concurrency-safe**
   - `esbuild.stop()` shuts down the shared esbuild service.
   - Multiple `Bundler` instances or overlapping builds can interfere with each other.
   - This is especially risky because `enableLiveReload(app)` starts an async bundle without awaiting it.

9. **Server-import guard is narrow**
   - It blocks only `.server.ts` and `.server.tsx`.
   - It does not catch `.server.js`, `.server.jsx`, `.server.mts`, extensionless imports, or query-suffixed imports if those are supported elsewhere.

10. **Remote endpoint path generation assumes local files under `Deno.cwd()`**
    - The client route hash uses `relative(Deno.cwd(), args.path)`.
    - If a `.remote.ts` file is outside cwd or passed as an unusual `file://`/plugin path, the generated client endpoint may not match the server route generated by `packRemoteFunctions`.

## Highest-confidence fixes

- Replace or scope the remote export detection so generated stubs match actual exports.
- Sort output files before generating the aggregate bundle hash.
- Avoid basename-only output mapping and classification.
- ~~Add `res.ok` and empty-body handling to generated remote function stubs.~~ ✅ Done — stubs now check `res.ok`, handle `204`, and detect unserializable returns.
- Make esbuild lifecycle ownership safer for overlapping builds.
